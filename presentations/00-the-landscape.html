Follow a single request end-to-end

GET /employees — the same way you'd trace Rails middleware

1 Route 53 DNS turns api.workforce.ensido.com into the ALB's address. lesson 09
2 ALB (public subnet) terminates HTTPS via an ACM cert, forwards inward to port 3000. lesson 08–09
3 ALB only forwards to containers that pass the health check GET /up — that's the target group. lesson 08
4 Request reaches a Rails container on ECS Fargate in a private subnet. No public IP — unreachable directly. lesson 07
5 Container receives traffic on port 3000 only from the ALB because a security group says so. lesson 04
6 Rails queries RDS Postgres and ElastiCache Redis — both in private subnets, app-only access. lessons 12–13

A Sidekiq job is the same path minus the ALB. Outbound calls go: private subnet → NAT Gateway → Internet Gateway. → lesson 03

Can a network packet travel from here to there on this port?

ALB → app :3000 ✓

Identity plane · lesson 05

IAM

Is this identity allowed to call this AWS API action on this resource?

task-role → s3:GetObject ✓
task-role → s3:DeleteBucket ✗

Your app can have perfect network access to S3 and still get AccessDenied — IAM said no. Or have IAM permission to read a secret and still time out — no network path. When something breaks: first question is always network or permission? App engineers constantly fix the wrong one.

Working with the agent

The agent has a terminal.
It inspects, it writes Terraform — you confirm before apply.

What the agent does for you

  • Inspects live AWS via CLI — aws ec2 describe-vpcs, describe-services — so it builds on reality
  • Writes & edits the .tf files for the change
  • Runs terraform plan and explains every line
  • Verifies after — re-describes the resource via CLI

The rule before every apply

  • It must explain: what this creates, cost idle/month, what breaks if a CIDR / SG / IAM rule is wrong
  • You confirm — explicitly. The agent never applies on its own
  • Can't explain a line? Don't apply.
  • Watch for insecure defaults — 0.0.0.0/0, public buckets, Action:*

The loop: Inspect → Propose → Explain → You confirm → Apply → Verify. Docs & the plan output outrank the agent's confidence — you own the architecture.
