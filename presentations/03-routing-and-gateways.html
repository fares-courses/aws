Routing &
Gateways

In lesson 02 you labeled subnets public and private — but that was intention, not fact. This is where the labels become behavior: route tables, the Internet Gateway that makes a subnet public, and the NAT Gateway that lets private tasks reach out without being reachable.

Mental model

The route table is a signpost
at every intersection.

"For the local network go here, for the internet go there." An Internet Gateway is a two-way street door (public). A NAT Gateway is a one-way exit — your private tasks can walk out to the internet, but nobody can walk in.

03
How a route table decides

Most-specific route wins

Every packet leaving a subnet is matched against its route table. AWS picks the longest-prefix match. One built-in local route (the whole VPC CIDR) can never be removed.

Route table · public subnet
DestinationTargetMeaning
10.0.0.0/16localbuilt-in: talk to anything in the VPC
0.0.0.0/0igw-xxxxeverything else → the internet

A packet to 10.0.5.10 matches local — stays internal. A packet to 52.x.x.x matches only 0.0.0.0/0 → IGW. That one line is the entire difference between public and private.

AWS always picks the longest prefix (biggest slash number). A /16 beats /0, so specific destinations win before the catch-all 0.0.0.0/0 ever applies.

Two gateways, two directions

IGW makes it public.
NAT makes it outbound-only.

Internet Gateway

The VPC's two-way door. One per VPC. A subnet goes public when its route table points 0.0.0.0/0 → IGW and the resource has a public IP.

Carries the ALB's inbound public traffic.

Free · no bandwidth limit

NAT Gateway

The one-way exit. Sits in a public subnet. Private subnets route 0.0.0.0/0 → NAT so tasks can reach out — but the internet can't reach in.

Pull images, deliver webhooks, call APIs.

~$32/mo + $0.045/GB 💸
The key asymmetry: a private task can initiate a connection outward and get the reply — but the internet cannot initiate a connection inward. That's how Sidekiq delivers a webhook while staying unreachable.

The most infamous AWS bill surprise

The NAT Gateway cost trap

A NAT Gateway costs two ways: ~$0.045/hr just for existing (≈$32/mo, even idle) and ~$0.045 per GB processed.

1 NAT GW, low traffic — Workforce prod today ~$32/mo
3 NAT GW (per-AZ), idle — often overkill ~$96/mo
Heavy S3-via-NAT egress — fix with a VPC endpoint (L14) $32 + $$$
Rule of thumb: one NAT is usually fine to start · add per-AZ NAT only when AZ-isolated egress failure is a real risk · route S3/ECR/Secrets through VPC endpoints, not the NAT.
07
Blast radius · the accidental exposure bug

One wrong line exposes everything

Someone adds this to the private subnets' route table "to fix a connectivity issue":

0.0.0.0/0 → igw-xxxx   ← WRONG target (IGW instead of NAT)
  • Private tasks with a public IP → now reachable from the internet
  • Security groups become the only thing between the internet and your app
  • You've silently converted "defense in depth" into "one misconfigured SG from breach"

The fix: private subnets route 0.0.0.0/0 to the NAT Gateway, never the IGW. This is why lesson 02 kept insisting: follow the route table.

The Terraform

It all reduces to one target

# Two-way public door — one IGW per VPC (free)
resource "aws_internet_gateway" "igw" { vpc_id = aws_vpc.main.id }

# PUBLIC route table — this single line makes subnets public
resource "aws_route_table" "public" {
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id   # → IGW = public
  }
}

# NAT lives in a PUBLIC subnet — the ~$32/mo component
resource "aws_nat_gateway" "nat" {
  allocation_id = aws_eip.nat.id
  subnet_id     = aws_subnet.public_a.id
}

# PRIVATE route table — → NAT, NOT the IGW. Outbound-only.
resource "aws_route_table" "private" {
  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.nat.id  # → NAT = outbound-only
  }
}

The whole public/private distinction is just which target 0.0.0.0/0 points at — IGW (two-way) or NAT (outbound-only). Everything else is wiring.

The Terraform · full annotated version

Gateways, route tables, associations

# Two-way public door: one IGW per VPC (free)
resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "main-igw" }
}

# PUBLIC route table: "everywhere" → IGW.
# This one route makes subnets public.
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }
  tags = { Name = "public-rt" }
}

# Associate public RT with each public subnet
resource "aws_route_table_association" "public_a" {
  subnet_id      = aws_subnet.public_a.id
  route_table_id = aws_route_table.public.id
}
resource "aws_route_table_association" "public_b" {
  subnet_id      = aws_subnet.public_b.id
  route_table_id = aws_route_table.public.id
}
# One-way exit: NAT in a PUBLIC subnet,
# needs a static EIP. ~$32/mo + per-GB.
resource "aws_eip" "nat" {
  domain = "vpc"
  tags   = { Name = "nat-eip" }
}
resource "aws_nat_gateway" "nat" {
  allocation_id = aws_eip.nat.id
  subnet_id     = aws_subnet.public_a.id
  tags          = { Name = "main-nat" }
  depends_on    = [aws_internet_gateway.igw]
}

# PRIVATE route table: "everywhere" → NAT,
# NOT the IGW. Outbound-only.
resource "aws_route_table" "private" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.nat.id
  }
  tags = { Name = "private-rt" }
}

# All private subnets share the one NAT
resource "aws_route_table_association" "private_a" {
  subnet_id      = aws_subnet.private_a.id
  route_table_id = aws_route_table.private.id
}
resource "aws_route_table_association" "private_b" {
  subnet_id      = aws_subnet.private_b.id
  route_table_id = aws_route_table.private.id
}
How Workforce does it · us-east-2

One NAT, not three — deliberately

The setup
  • 1 Internet Gateway — the public door
  • 1 NAT Gateway + 1 Elastic IP, in a public subnet
  • 0.0.0.0/0 → IGW makes public subnets public
  • 3 private routes 0.0.0.0/0 → NAT — all AZs share the one NAT
The tradeoff to explain

One NAT (~$32/mo) vs one-per-AZ (~$96/mo). If the NAT's AZ fails, outbound internet from other AZs breaks — but inbound via the multi-AZ ALB still works.

A deliberate cost tradeoff, not an oversight.

The Workforce Agent environment drops the NAT entirely and uses VPC endpoints (S3 + ECR + Secrets + logs) instead — same problem, cheaper solution. You'll fully get it after lesson 14.
