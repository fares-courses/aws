03
What a VPC actually isolates

Network packets. Not API calls.

✓ What it isolates

  • Network traffic — no packet from outside arrives without a configured path
  • IP address space — your 10.0.x.x addresses are meaningful only inside this VPC

✗ What it does NOT isolate

  • IAM / API access — an identity with s3:GetObject can call it from inside any VPC or none
  • Billing / account — a networking construct, not a cost boundary
Two separate planes: a VPC blocks packets, IAM blocks API calls. You can fail one and pass the other. Don't reason "it's inside the VPC, so it's safe" — that only applies to network traffic.
Reading CIDR blocks without fear

Smaller slash = bigger network.

/N means the first N bits are locked. You get 2(32−N) addresses.

CIDRAddressesPlain English
10.0.0.0/1665 536"I own all of 10.0.x.x" — this is your VPC
10.0.0.0/24256"I own all of 10.0.0.x" — one subnet
10.0.1.0/24256"I own all of 10.0.1.x" — another subnet
10.0.0.0/2816A tiny slice — too small for most things

A /16 VPC can hold 256 non-overlapping /24 subnets. That nesting is exactly how you'll carve rooms next lesson. You never compute this by hand — but you must be able to read it.

05
The one permanent decision

Pick a range that won't collide
with anything you'll ever connect to.

Use the private ranges — 10.x, 172.16–31.x, 192.168.x. But more importantly: pick a non-obvious slice per environment.

Prod
10.20.0.0/16
Staging
10.30.0.0/16
Dev
10.40.0.0/16

Never use 172.31.0.0/16 — that's the AWS default VPC range, identical in every account. If you ever peer, VPN, or connect a partner network, you can't route because the address is ambiguous. The CIDR is effectively permanent — you can add secondary blocks but can't relocate the primary without rebuilding.

The checkbox you always forget

Always turn on DNS hostnames.

enable_dns_support

Lets resources resolve DNS at all (AWS resolver). Default on. Leave it.

enable_dns_hostnames

Gives resources internal DNS names. Default OFF on custom VPCs.

Always set this to true.

Without it: RDS gives you a hostname like mydb.xyz.us-east-1.rds.amazonaws.com that your app can't resolve. The failure looks like a random network error — but it's just a DNS flag. Turn it on in every VPC and never think about it again.

The Terraform

A bare VPC is a sealed box — everything else is deliberate.

resource "aws_vpc" "main" {
  cidr_block           = "10.20.0.0/16"   # §3.2 — floor space; deliberate non-default slice

  enable_dns_support   = true            # on by default — keep it
  enable_dns_hostnames = true            # OFF by default — always flip to true

  instance_tenancy     = "default"       # shared hardware; 'dedicated' is $$ for compliance

  tags = {
    Name        = "prod-vpc"
    Environment = "production"
  }
}

No subnets, no internet access, no firewall rules. A bare VPC is sealed by design. Everything that follows is you deliberately opening it. The CIDR is the one value you supply — never let the agent pick it.

Footguns

Common mistakes, crisp fixes

!Sizing too small — a /24 VPC runs out of IPs as Fargate tasks each consume one. Default to /16 (it's free).
!Using the default VPC (172.31) — same range in every account, public subnets by default. Always make your own.
!Forgetting dns_hostnames — RDS/ElastiCache names won't resolve; you'll chase a phantom network bug.
!"It's inside the VPC so it's safe" — VPC only bounds packets. IAM is separate. Two different planes.
!Same CIDR for prod + staging — works fine in isolation, impossible to route between them later.
